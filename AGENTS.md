# AI_Prompt

Kilo Code AI Agent 项目级行为约束与编码规范。本文件为永久性约束，适用于项目中所有 AI Agent 会话。

## 核心约束

1. 当用户提出需求时，先分析拆解需求，并将理解列点回馈给用户确认。如果需求极其简单且不存在歧义（例如查询当前时间、简单计算等），可以跳过确认直接执行，但仍需在响应中简要说明你的理解。分析过程中遇到不确定的内容必须及时提问，避免推测性假设。

2. 设计应保持简洁，避免过度设计。执行前需判断设计方案是否会引入不必要的复杂度。若出现以下任一情况，则视为可能过度设计，必须与用户沟通确认后再继续：
    - 新增或修改的文件超过 3 个
    - 引入新的抽象层（如新增基类、中间件、设计模式包装）但无明显复用需求
    - 为单一功能引入第三方库或框架
    - 计划中包含过多未来扩展点（基于当前尚未确认的需求）
      若用户明确要求简洁实现，以上阈值自动降低。

3. 严格控制修改范围，避免修改与当前需求无直接关系的既有代码（即使它们在同一个文件或相邻逻辑块中）。小规模重构（如重命名变量、提取方法）必须事先告知用户，说明原因和影响范围。大规模重构或架构变更必须获得用户明确同意后才能进行。整体开发应遵循已制定的开发规划，避免局部频繁、零散的修改破坏一致性。

4. 当需求包含多步骤操作、存在多种同等合理的技术方案，或需求本身较为模糊时，必须主动列出可选方案及各自的优劣，让用户进行选择确认。禁止在未取得用户确认的情况下直接选择实施路径。

5. 完成代码修改后应及时运行相关测试，验证功能正确性并确认未引入回归问题。测试未通过前不得声称任务完成，必须先排查并修复失败用例。若项目缺少测试框架或测试用例，应主动向用户建议补充，以便后续自动化验证。

6. （元规则）当以上约束彼此冲突，或约束与用户明确指令冲突时，优先级为：用户明确指令 > 安全性/数据完整性 > 本文件中的其他约束。发生冲突时，Agent 必须向用户报告冲突点及采取的仲裁策略。若 Agent 发现自身已违反某条规则，应立即说明并纠正。

## 操作规范

- 会话开始时，检查并设置全局默认编码为 UTF-8（针对命令行工具、脚本文件读写等）。写入任何文件时需显式指定 UTF-8 编码。如果使用shell工具，需要在每次启动时设置编码为 UTF-8。
- 代码标识符（变量、函数、类名等）使用英文。代码注释使用中文（解释意图、待办事项、复杂逻辑）。文档使用中文，遵循项目既有约定。向用户展示的分析说明文本可用中文。
- 对话及思考中除了专有名词或约定成俗的名词外，其他名称应使用中文。
- 单个文件内容不宜过多。当文件满足以下任一条件时，应考虑拆分为子文件并建立索引，父文件仅保留简述和可跳转的链接：
    - 行数超过 200 行
    - 包含超过 5 个明显可独立描述的功能模块
    - 文件包含多个不同抽象层次的内容（如同时包含高层流程描述和底层实现细节）
- 如果发生计划外操作或与计划产生偏差，必须在操作前向用户说明并寻求确认，同时在日志中记录偏差及原因。
- 每次代码修改完成后，同步更新相关文档（注释、README、API 文档等），确保文档与代码保持一致，不产生过期信息。同步范围包含 `.ai/` 工作区内的计划、日志、笔记等文件，保持工作区记录与实际代码状态一致。
- 如果能够使用版本管理，尽量使用版本管理。具体选择需要根据项目情况给用户提出建议，让用户决定。

## 编码风格

- 使用早返回模式降低嵌套深度，避免超过 3 层嵌套。
- 避免无意义的 `else` —— `if` 块已经 `return` 时直接走后续逻辑。
- 条件/循环体即使只有一行也必须使用大括号。
- 空值检查优先使用早返回或空值合并，避免深层 null 判断嵌套。
- 异步操作优先使用语言的 async/await 模式。
- 优先使用不可变声明（`const`），减少副作用。
- 指针/空值比较使用 `nullptr`（C++），不使用 `NULL` 或 `0`。

## 注释规范

- 类/结构体/枚举：声明处必须有中文注释说明职责和用途。
- 公共方法/属性：必须有中文注释说明功能、参数含义、返回值。
- 重要逻辑分支/生命周期/状态机：必须有一行中文注释解释意图。
- 错误路径：每个错误返回前应有中文注释说明触发条件。
- 关键文件头部需中文注释说明文件职责。

## 动态规则

项目运行中产生的具体规则沉淀在 `.ai/dev/dev_core.md` 中，使用 `[+]` / `[-]` 标记管理启用/禁用状态。该文件优先级高于本文件，但低于用户的直接指令。

## AGENTS.md 写入规范

本文件为 AI Agent 行为准则，在 Kilo 等工具中受写入保护。当 AI Agent 需要沉淀新规则时，按以下层级判断写入位置：

- **项目根目录 `AGENTS.md`**：适用于全局性约束、项目整体规范、跨模块通用的编码约定。
- **子目录 `AGENTS.md`**：适用于仅对该目录或其子目录生效的规则（如 `src/backend/AGENTS.md` 存放后端专用规范）。子目录规则与根目录规则叠加生效，冲突时子目录规则优先。
- **`.ai/dev/dev_core.md`**：适用于频繁变动的、探索性的临时规则，通过 `[+]` / `[-]` 开关管理，不与 AGENTS.md 混淆。

判断原则：规则影响范围越广，写入层级越高；规则越稳定，越适合放入 AGENTS.md 而非 dev_core.md。

## 知识库

关于 `.ai/kb/` 项目知识库的写入规范、分类结构与索引维护，参见 `Kilo/Instructions/kilo_instructions_core.md` 中的 `## 知识库` 章节。

**核心区分原则**：
- **约束体系**（AGENTS.md / Instructions / dev_core.md）：记录"应该怎么做"。
- **知识库**（.ai/kb/）：记录"这个项目是什么样的"和"遇到问题怎么办"。
- 调试流程、编译流程、架构决策、代码模式 → 写入知识库，不写入 AGENTS.md。
